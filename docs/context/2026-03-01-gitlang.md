## Architecture Notes

### Input Syntax
- `user` — all repos for user
- `user/repo` — single repo
- `user/repo1,repo2` — multiple repos
- `user1 user2` or `user1+user2` — compile (aggregate)
- `user1 ? user2` — compare (side-by-side)
- `user/repo@author` — contributor's commit-level language breakdown
- `@user` or `org:name` — explicit user/org prefix

### Key Files
- `server/helpers/github/tokenManager.ts` — multi-token rotation with rate-limit awareness
- `server/helpers/github/contributors.ts` — commit-level language analysis via file extension mapping
- `server/helpers/github/rateLimit.ts` — rate limit info endpoint
- `server/helpers/github/auth.ts` — AWS Secrets Manager with 5-min TTL cache
- `client/lib/data/index.ts` — data pipeline: parsing → fetching → aggregation → breakdown
- `infra/lambda/rotate-token.ts` — Lambda function for Secrets Manager rotation
- `infra/cloudformation-rotation.yml` — CloudFormation deployment template

### Data Flow
1. Input parsed by `parseEntry()` → `{owner, repos, author}`
2. Groups split by `~` (compare) or processed as single group
3. Within groups, entries split by `+` (compile/aggregate)
4. Per entry: fetch repos → fetch langs (or contributor langs if @author) → aggregate with `getSize()`
5. `buildLangRepoBreakdown()` preserves per-repo attribution for expandable details
